Inside every Git repository is a hole.

`.git/objects/` — a directory where nothing is ever edited. Only added. Written once, never touched again. A place where a file's content becomes its name.

Run this:

```bash
$ echo "hello" | git hash-object --stdin
22596363b3de40b06f981fb85dac12e3f1c0f3d5
```

Now run it again. Same output. Give that exact string to anyone on any machine anywhere, and they get the same 40-character hash. The hash *is* the content. The content *is* the hash. They are the same thing, expressed two ways.

That is the foundation of everything Git does. Let's see where it leads.

## What Lives in `.git/objects/`

Start a new repository. Make a file. Commit it. Then look:

```
.git/
  objects/
    3b/
      18e512dba79e4c8300dd08aeb37f8e728b8dad
    e6/
      9de29bb2d1d6434b8b29ae775ad8c2e48c5391
    ab/
      c123... (and so on)
```

Those are your objects. Stored in subdirectories named after the first two hex characters of their hash. The remaining 38 characters are the filename. This is not random — it's a content-addressable filesystem.

There are exactly three kinds of objects. No more. A blob is the simplest. A tree describes a directory. A commit exists to remember who did what when.

## Blobs

A blob is just the bytes of a file. Not the name. Not the permissions. Not the line endings. Just the bits.

```bash
$ echo "hello world" > README.md
$ git hash-object README.md
3b18e512dba79e4c8300dd08aeb37f8e728b8dad
```

Now rename the file:

```bash
$ mv README.md NOTE.md
$ git hash-object NOTE.md
3b18e512dba79e4c8300dd08aeb37f8e728b8dad
```

Same. The content did not change, so the hash did not change. The name is irrelevant. The blob does not know it was in a file called README.md. It does not know it is now in NOTE.md. It only knows its bytes.

You can look at what Git actually stored:

```bash
$ git cat-file -p 3b18e512dba79e4c8300dd08aeb37f8e728b8dad
hello world
```

## Trees

Files live in directories. Directories have names. A tree is a mapping from names to hashes.

```
100644 blob 3b18e51    README.md
040000 tree a1b2c34    src/
```

Each line says: this entry has these Unix permissions, it points to an object of this type with this hash, and here's the name. That's a directory.

The tree itself becomes an object. Its hash is computed from its own content — the sorted list of entries. If you rename a file inside a tree, the tree's hash changes. If you change a file's content (and thus its blob hash), the tree's hash changes. The tree is a commitment to a particular arrangement of names and content.

You can inspect any tree:

```bash
$ git cat-file -p HEAD^{tree}
100644 blob 3b18e51    README.md
040000 tree a1b2c34    src/
```

## Commits

A commit is a small text object that says: "At this point, the project had the state described by this tree."

```
tree a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0
parent c9d0e1f2a3b4c5d6e7f8a9b0a1b2c3d4e5f6a7b8
author Bernard <adjanour@icloud.com> 1747000000
committer Bernard <adjanour@icloud.com> 1747000000

Add documentation for the config parser
```

The commit points to a tree. It points to zero or more parents. It records the author, the committer, a timestamp, and a message. Hash all of that together and you get the commit's identifier.

This is the chain. Every commit remembers its parents. The parents remember theirs. Trace backwards and you reach the root — the first commit, which has no parent.

## The Chain Reaction

Because every object's identity is its content, the objects are all bound together. Change one byte in a file. The blob hash changes. The tree that referenced that blob now has a different entry — its hash changes. The commit that referenced that tree now points to a different tree — its hash changes. Every subsequent commit that chains from that commit changes too.

One byte. Ripples through the entire history.

This is not a failsafe or a nice property. It is the mechanism. It is how Git knows whether anyone has tampered with history. It is how two clones can compare their entire histories by exchanging a single 20-byte hash.

```bash
$ git log --oneline
a1b2c3d Add documentation
d4e5f6a Implement config parser
g7h8i9b Initial commit
```

What the output doesn't show: each of those hashes commits to everything before it.

```
a1b2c3d → tree:xyz, parent:d4e5f6a
d4e5f6a → tree:uvw, parent:g7h8i9b
g7h8i9b → tree:rst, parent:(none)
```

The root commit is the only one without a parent. Everything else traces back. If you change something in g7h8i9b's tree, d4e5f6a's hash changes. Which means a1b2c3d's hash changes. The entire chain moves.

## Plumbing

Git's user-friendly commands are called *porcelain*. The low-level commands are *plumbing*. Once you know the object model, the plumbing makes sense:

| Command | What it does |
|---------|-------------|
| `git hash-object` | Hash a file and optionally store the blob |
| `git cat-file -p` | Print the contents of any object |
| `git ls-tree` | List entries in a tree |
| `git write-tree` | Create a tree object from the staging area |
| `git commit-tree` | Create a commit object from a tree |
| `git update-ref` | Move a branch pointer to a different commit |

These are the primitives. Porcelain commands call plumbing commands. `git add` calls `hash-object` and `update-index`. `git commit` calls `write-tree` and `commit-tree`. There is no magic — the higher commands are just scripts that string the low-level ones together.

You can prove this to yourself. Make a commit without using `git commit`:

```bash
$ echo "hello" | git hash-object -w --stdin
22596363b3de40b06f981fb85dac12e3f1c0f3d5

$ git update-index --add --cacheinfo 100644 \
    22596363b3de40b06f981fb85dac12e3f1c0f3d5 hello.txt

$ tree_hash=$(git write-tree)

$ commit_hash=$(echo "my commit" | git commit-tree $tree_hash)

$ git update-ref refs/heads/main $commit_hash

$ git log --oneline
a1b2c3d my commit
```

No `git add`. No `git commit`. Yet the commit exists. The tree exists. The blob exists. Git does not care how the objects got there. It only cares that they are there, named by what they contain.

## What This Means

Content addressing creates a universal namespace. You and I, on different machines, in different countries, can create files with the same content and get the same hash. We can create trees with identical entries and get the same hash. We can craft commits with identical metadata — tree, parent, message, timestamps — and get the same hash.

Git does not store files. It stores objects. A filename is just a piece of metadata in a tree object. A file's *content* is its true identity. This is why Git detects renames automatically. This is why identical files across a repository share a single blob. This is why clones are cryptographic backups: every hash is verifiable, every chain traceable.

The next time you commit, consider what actually happens. Your files become blobs. Your directory structure becomes a tree. A small text object — the commit — records the tree hash, the parent hash, your name, and your message. A branch pointer moves. Nothing is ever edited. Everything is added.

The object store accumulates. It never forgets.
